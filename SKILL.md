# Rule Evolution (规则进化与经验沉淀)

本技能旨在帮助 AI 代理在复杂、多轮的对话结束后，将当前会话中探索出的物理规律、排灾步骤、接口设计偏好等高价值经验，自动沉淀并固化到全局开发规范文件 [AGENTS.md](file:///Users/cheng/.gemini/config/AGENTS.md) 中，实现 Agent 规则的“自我进化”。

## 🎯 适用场景 (When to Use)

- 当一个复杂的 Bug 被成功排查并修复后（例如解决了网络冲突、死锁、特定客户端兼容性问题）。
- 当与用户商定并实现了一套新的架构设计、API 规范或环境配置方案时。
- 当对话中经历了“多次报错 -> 调试 -> 成功解决”的曲折路径时，Agent 应当**主动提示用户**是否需要沉淀规则。
- 当用户明确发出指令，要求“将刚才解决的问题总结并更新到规则中”时。

## ⚙️ 核心工作流 (Core Workflow)

AI 代理在加载本技能后，必须严格按照以下五个步骤执行：

### 第一步：Review 历史与提取经验 (Extract Experience)
1. **分析上下文**：使用 `view_file` 或相关工具查看当前会话的完整历史或 `transcript.jsonl` 日志，梳理解决过的问题、做过的关键代码修改（Diff）以及配置调整。
2. **双向经验提炼**：将这些具体的改动提炼为**通用的、面向未来开发的**规范：
   - **正面指导原则**：应该怎么做，如何规范设计。
   - **血泪负面案例 (Anti-Patterns)**：绝对不能怎么做，记录当时断网/死锁的具体物理表现和导致原因。
   * *负面案例范例*：“在 Merge.yaml 中排除了 10.58.131.125，但漏掉了 10.0.0.0/8。后果：导致 Tun 模式下其余内网域名走代理，被公司 WAF 直接拦截而断网。”

### 第二步：对比去重与防冲突 (Conflict & Duplicate Check)
1. **读取现有的 `AGENTS.md`**：读取项目及系统当前的 [AGENTS.md](file:///Users/cheng/.gemini/config/AGENTS.md)（包括链接的真实目标文件）。
2. **比对排查**：
   - **去重**：如果提炼出的经验在 `AGENTS.md` 中已有等价或更通俗的表述，则直接忽略。
   - **防冲突**：如果有新规则与旧规则相违背，必须停下来与用户沟通，阐明冲突点，获得授权后方可修改旧规则。

### 第三步：自动备份与版本回滚防护 (Pre-Write Protection)
1. **本地备份**：在对 [AGENTS.md](file:///Users/cheng/.gemini/config/AGENTS.md)进行任何写入前，先在同目录下拷贝一份临时备份文件（如 `AGENTS.md.bak`），确保在写入出错时可一键恢复。
2. **Git 版本标记**：在完成文件写入后，如果当前项目在 Git 版本控制中，建议在终端执行 Git 提交以保留版本追溯：
   ```bash
   git add <agents.md_path> && git commit -m "chore(agents): 自动规则沉淀 - [新增规约简述]"
   ```

### 第四步：按既定格式与草案归类整理 (Formatting & Graduation)
按照 [AGENTS.md](file:///Users/cheng/.gemini/config/AGENTS.md) 现有的排版风格组织新条目：
* 语言必须使用**规范、精炼的中文**。
* 每一个子条款采用 `* **[核心关键词]**：[具体规则与避坑说明]` 的句式。
* **草案孵化设计**：新加入的散落规则或未经多项目验证的经验，优先追加在相应的**【规则扩展锚点】**中作为“草案规则 (Draft Rules)”。当该规则在后续开发中被反复验证成熟，或者在定期重构中，再将其移入核心板块。

### 第五步：安全更新与持久化验证 (Safe Update & Verify)
1. **精准更新**：修改文件（如存在软链接，确保修改应用到链接的真实源文件中）。
2. **结果验证**：更新完毕后，使用 `cat` 或 `grep` 验证写入结果，并删除临时备份 `.bak` 文件。
